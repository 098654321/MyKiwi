## 实验

##### 现在的算法思路

- 直接把其余 mode 的 net 都加载进来，在不影响当前 mode 布通性的情况下直接一起布线

- 单独被可以复用的 net 占用的寄存器是不需要修改的，含有不可复用的 net 的寄存器其大概率要被修改。大概率是指在切换的时候这些 net 不再使用，所以其用到的寄存器需要修改，但是有可能遇到另一个 net 同样使用了这些寄存器，所以也有可能不修改。根据这一点，可以产生两种思路：

    1. 在当前 mode 布线的时候，让不可复用的 net 使用的寄存器尽量少，这样在切换 mode 的时候，需要清除的寄存器就少

    2. **在下一个 mode 布线的时候，那些不可复用的 net 尽量走上一个 mode 里面不可复用的 net 使用过的寄存器，减少修改**

- 目前考虑第 1 步。之前的算法目的是把两种类型的 net 分开，但是这样还不够，更需要把不可复用的 net 聚到同一个寄存器里面

- 修改思路和评价指标之后，使用包含不可复用 net 的寄存器数量来衡量。这还是一个间接的指标，好处是可以每跑完一轮迭代就反应相关的信息（因为两个 mode 的寄存器配置信息对比需要完成整个布线过程才能得到）

    * 这里的实验发现**重布线是导致被不可复用 net 占用寄存器数量增加的主要原因。因为重布线会导致总线的长度变长**。

    * 数据分析发现 nonreuse 和最终修改的寄存器数量之间线性相关性很弱

- **最后把评价指标修改成直接看两个模式的配置文件当中不同的寄存器数量**

    * 设计一组数据，验证迭代过程的效果，并且排除同步总线的重布线过程对于最终相同/不同寄存器数量的影响，是的最终统计的数据仅依赖于迭代本身:

        [](./case17.png).每个 TOB 有两个 bank ，每个 bank 里面的同一排 bump 相互竞争 mux 资源，一定会被分到不同组里面。但是不同排之间的bump 不会竞争 mux 资源，可以自由选择分配到哪一组。所以可以每个 bank 的 8 排里面每一排使用一个 bump 。

    * 从思路上看，这个算法在单个轮次内的布线质量受布线顺序的影响较大，因为前面布线的结果会影响后面布线的代价，这和 PathFinder 的问题是一致的

    * 之前的算法在更新 track 的时候是以 reg 的 bit 为单位，根据当前占用数量和历史使用情况来分析每个 bit 应该分配给那种类型的 net 。但是最终考虑的是整个寄存器的情况，所以在更新当前使用情况的时候应该要考虑整个寄存器的情况，一个寄存器里面的 bit 一起更新代价。在更新历史信息的时候还是以 bit 为单位记录每个 bit 被两种不同类型 net 使用的历史记录，从切换的角度，一个新模式使用的资源需要跟上一个模式完全重合，不只是位于同一个寄存器，也需要占用同一个寄存器里面的相同的 bit ，所以需要以 bit 为单位记录细节的信息。

    * 布线路径会受到附近其它同类型线网的路径的影响，尤其是长的线会收到附近短的线的影响，在迭代的后期被这些路径拉过去，导致走的不一定是最短的路径。

    * 路径在相同寄存器里面的聚集也代表了资源冲突与争夺加剧，导致当前较短的路径无法连通，当另一条长度更长但是没有资源争夺的路径出现的时候，有可能这条路径因为能连通而被优先选择，导致路径变长。

    * 当前版本的代码里面布通率和 version_before_commands 比有下降，为什么？

    ```c++
    cycle = 0
    load nets, sort nets by reuse frequency（用 reuse frequency 排序）
    while (true) {
        for net in nets {
            path = route net with maze(using the new cost function)
        }
        
        if fail {
            if (cycle == 0) {
                remove nets in other modes
                reroute
            }
            else {
                return success with path from last cycle
            }
        }
            
        update recorder        

        if (cycle >= MIN_CYCLE_NUMBER) {
            return success with path
        }

        cycle++
    }
    ```

- 上面的那个算法有个问题，就是在当前 mode 就加载了所有 mode 的 net 并布线，这样会相互挤占布线资源，导致布线质量下降，例如路径相互阻碍导致需要绕路使长度变长。原来是希望一次把两个 mode 的 net 布完，这样在给当前 mode 布线的时候就不会导致当前 mode 布线的时候不考虑后面的情况，让后面一个 mode 的 net 布线失败。同时两个 mode 的需要切换的 net 的路径可以提前凑到一起，减少需要切换的 net 使用的寄存器总数。

    还可以再加一个策略，当前 mode 布线之前把已有的 mode 的布线结果加载进来（前提是已有的 mode 有布线结果了），然后共用的线直接用已有的路径，不重新布线，并利用已有的路径更新代价值。
